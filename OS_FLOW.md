# OpenSpec CLI — как это работает

Гайд для того, кто **никогда** не работал с OpenSpec. Простыми словами: что это, из чего состоит и как этим пользоваться шаг за шагом.

---

## 1. Какую задачу решает

Когда над кодом работают люди и AI-агенты, возникают две вечные проблемы:

1. **Спека расходится с кодом.** Документация «что система должна делать» устаревает через неделю после написания.
2. **Агент не знает контекста.** Чтобы AI сделал изменение правильно, ему нужно скормить актуальное описание системы и понятный план — каждый раз вручную.

OpenSpec решает это так:

- Держит **живую спеку** системы в репозитории (в папке `openspec/`) — это «источник правды» о том, что система делает **сейчас**.
- Любое изменение сначала оформляется как **change** (предложение) — набор понятных документов: *что меняем, зачем, как и какими шагами*. Их можно прочитать и обсудить **до** написания кода.
- CLI «AI-native»: он умеет отдавать агенту готовые **инструкции и шаблоны** для каждого документа, а также **проверять** результат. То есть OpenSpec — это рельсы, по которым едет и человек, и AI.

> Коротко: **спека → предложение изменения → реализация → влить изменение обратно в спеку.** По кругу.

## 2. Мысленная модель (одна картинка)

```
                       openspec/
                          │
        ┌─────────────────┼──────────────────┐
        │                 │                  │
     specs/            changes/           config.yaml
   «как есть           «что хотим          (схема, контекст
    сейчас»             изменить»           проекта, правила)
        ▲                 │
        │                 │  когда изменение готово и внедрено
        └─────────────────┘  дельта вливается обратно в specs/
             archive           (change уезжает в changes/archive/)
```

- **`specs/`** — правда о текущем поведении системы.
- **`changes/`** — предложения по изменению этой правды (одно предложение = одна папка).
- Когда change реализован — его правки **вливаются в `specs/`**, а сам change **архивируется**.

## 3. Понятия и артефакты

| Термин | Простыми словами |
|---|---|
| **Capability** (способность) | Кусок функциональности системы. Пример: `auth`, `billing`, `http-server`. В `specs/` это отдельная папка. |
| **Requirement** (требование) | Одно правило поведения. Пишется как `### Requirement: ...` и обычно содержит слово **SHALL** («система ДОЛЖНА …»). |
| **Scenario** (сценарий) | Конкретный пример работы требования в формате `#### Scenario:` с **WHEN / THEN** («когда … тогда …»). У требования должен быть хотя бы один сценарий. |
| **Spec** (спека) | Файл `specs/<capability>/spec.md` — набор требований и сценариев одной способности. |
| **Change** (изменение) | Папка `changes/<имя>/` — предложение изменить систему. Внутри — несколько документов (артефактов). |
| **Delta** (дельта) | То, что change **предлагает** поменять в спеке: секции `## ADDED / MODIFIED / REMOVED / RENAMED Requirements`. Это как diff, но для требований. |
| **Artifact** (артефакт) | Один документ change-а: `proposal.md`, `design.md`, дельта-спеки, `tasks.md`. |
| **Schema** (схема) | Рабочий процесс — какие артефакты и в каком порядке. По умолчанию `spec-driven`: **proposal → specs → design → tasks**. |
| **Root** (корень) | Папка `openspec/`, с которой работает CLI. По умолчанию берётся ближайшая вверх по дереву от текущей директории. |

### Как выглядит папка openspec на диске

```
openspec/
├── config.yaml                 # схема + контекст проекта + правила
├── specs/                      # ЖИВАЯ спека («как есть сейчас»)
│   └── auth/
│       └── spec.md
└── changes/                    # активные предложения
    ├── add-two-factor/
    │   ├── .openspec.yaml       # метаданные change-а (какая схема)
    │   ├── proposal.md          # ЧТО и ЗАЧЕМ
    │   ├── design.md            # КАК (техническое решение)
    │   ├── specs/               # ДЕЛЬТА: как меняются требования
    │   │   └── auth/
    │   │       └── spec.md
    │   └── tasks.md             # чеклист реализации
    └── archive/                 # завершённые change-ы
        └── 2026-07-22-add-login/
```

### Как выглядит содержимое (мини-примеры)

**Требование в основной спеке** (`specs/auth/spec.md`):

```markdown
## Purpose
Пользователи входят в систему по email и паролю.

## Requirements

### Requirement: Login
The system SHALL authenticate a user by email and password.

#### Scenario: Valid credentials
- **WHEN** пользователь вводит корректные email и пароль
- **THEN** система создаёт сессию и пускает внутрь
```

**Дельта в change-е** (`changes/add-two-factor/specs/auth/spec.md`) — только то, что меняется:

```markdown
## ADDED Requirements

### Requirement: Two-factor code
The system SHALL require a one-time code after a successful password check.

#### Scenario: Code required
- **WHEN** пароль верный
- **THEN** система просит 6-значный код из приложения-аутентификатора
```

**Задачи** (`tasks.md`) — обычный markdown-чеклист:

```markdown
## 1. Backend
- [ ] Добавить генерацию TOTP-секрета
- [x] Хранить секрет у пользователя

## 2. UI
- [ ] Экран ввода кода
```

## 4. Основной флоу (spec-driven), шаг за шагом

Это стандартный цикл по умолчанию. Ниже — команды и что происходит на каждом шаге.

### Шаг 0 — Инициализация (один раз на проект)

```bash
openspec init
```

Разворачивает структуру `openspec/` (`config.yaml`, пустые `specs/` и `changes/`). После этого можно заполнить в `config.yaml` блок `context:` — короткое описание проекта (стек, домен, конвенции), которое CLI будет подсовывать агенту.

> Обновить служебные инструкционные файлы позже: `openspec update`.

### Шаг 1 — (по желанию) Подумать

Прежде чем формализовать, полезно просто «поразмышлять» над идеей — это делают в диалоге с агентом (в opencode это `/opsx-explore`). На уровне CLI отдельной команды нет; шаг опциональный.

### Шаг 2 — Создать change

```bash
openspec new change add-two-factor
```

Создаёт папку `changes/add-two-factor/` с метаданными `.openspec.yaml`. **Сами документы (proposal/design/specs/tasks) ещё пустые** — их создают по одному на следующих шагах.

### Шаг 3 — Спросить «что делать дальше»

```bash
openspec status --change add-two-factor
```

Показывает **граф артефактов** и их статус:

- `ready` — можно создавать (зависимости готовы),
- `blocked` — ждёт другой артефакт,
- `done` — готов.

А ещё: `applyRequires` (что обязательно нужно перед реализацией) и `nextSteps` (подсказка следующей команды). Добавьте `--json`, если результат читает скрипт или агент.

### Шаг 4 — Получить инструкции и написать каждый артефакт

Для каждого артефакта, который стал `ready`:

```bash
openspec instructions proposal --change add-two-factor
```

Команда отдаёт **всё, что нужно, чтобы написать документ**: шаблон (`template`), контекст проекта, правила, путь для записи и какие уже готовые артефакты прочитать для контекста. Дальше вы (или агент) пишете файл по шаблону и снова вызываете `status`, пока не будут готовы все обязательные артефакты.

Порядок по умолчанию: **proposal → specs (дельты) → design → tasks**.

### Шаг 5 — Проверить корректность

```bash
openspec validate add-two-factor --strict
```

Проверяет, что change оформлен правильно: есть дельты, у каждого требования есть хотя бы один `#### Scenario:` и т.д. Если что-то не так — покажет понятные ошибки. `--json` — машиночитаемо.

### Шаг 6 — Реализовать задачи

Идёте по `tasks.md` и пишете код. По мере готовности отмечаете пункты `- [ ]` → `- [x]`. (В opencode это `/opsx-apply` — агент реализует задачи и сам проставляет галочки.)

### Шаг 7 — Заархивировать (и влить дельту в спеку)

```bash
openspec archive add-two-factor
```

Переносит change в `changes/archive/YYYY-MM-DD-<имя>/` и **вливает его дельты в основные `specs/`** — теперь живая спека снова отражает реальность. Круг замкнулся.

### Весь цикл на одной схеме

```
init ──▶ new change ──▶ status ──▶ instructions (по каждому артефакту)
                                          │
                                          ▼
                          proposal · specs(дельта) · design · tasks
                                          │
                                          ▼
                          validate ──▶ реализация задач ──▶ archive
                                                                │
                                                    дельта вливается в specs/
```

## 5. Ежедневные команды (шпаргалка)

| Хочу… | Команда |
|---|---|
| Список активных changes | `openspec list` |
| Список спек | `openspec list --specs` |
| Посмотреть change/спеку | `openspec show <имя>` |
| Только дельты change-а | `openspec show <имя> --json --deltas-only` |
| Статус артефактов change-а | `openspec status --change <имя>` |
| Проверить всё разом | `openspec validate --all --strict` |
| Интерактивный дашборд | `openspec view` |

> Почти у всех команд есть `--json` — это «режим для скриптов и агентов»: тот же ответ, но структурой, а не текстом.

## 6. Дополнительные возможности

Их можно не трогать на старте — они нужны, когда проект растёт.

### Свой рабочий процесс — Schemas

По умолчанию используется схема `spec-driven`. Если ваш процесс другой (например, `spec → tests → implementation → docs`), можно сделать свою схему:

```bash
openspec schemas                 # список доступных схем
openspec schema fork spec-driven my-flow   # скопировать и настроить под себя
openspec templates --schema my-flow        # посмотреть шаблоны артефактов
```

### Несколько репозиториев спек — Stores

**Store** — это отдельный, самостоятельный OpenSpec-репозиторий, зарегистрированный на вашей машине (например, общая спека платформы, которую переиспользуют разные проекты).

```bash
openspec store setup platform-specs    # создать и зарегистрировать
openspec store list                    # что зарегистрировано
openspec list --store platform-specs   # работать с конкретным store
```

Флаг `--store <id>` понимают команды, которые читают/пишут спеки и changes (`list`, `status`, `show`, `validate`, `archive`, `instructions`, `new change`, `context`, `doctor`).

### Личные наборы папок — Worksets

**Workset** — сохранённый личный «рабочий вид»: набор папок, который удобно открыть разом в редакторе или в сессии агента. Ничего в самих папках не меняет.

```bash
openspec workset create my-view    # собрать и сохранить набор
openspec workset open my-view      # открыть его
```

### Настройки — Config

```bash
openspec config list          # текущие настройки
openspec config set <key> <value>
openspec config profile       # выбрать профиль рабочего процесса
openspec config edit          # открыть конфиг в $EDITOR
```

Главное в `openspec/config.yaml`:
- `schema:` — какой процесс используется,
- `context:` — описание проекта для агента,
- `rules:` — доп. правила для конкретных артефактов (например, «proposal ≤ 500 слов»).

### Диагностика — doctor и context

```bash
openspec doctor     # проверить «здоровье» корня и связей
openspec context    # краткий бриф рабочего контекста (можно --json)
```

### Мелочи

```bash
openspec feedback "текст"    # отправить фидбэк о самом OpenSpec
openspec completion          # автодополнение команд в шелле
openspec update              # обновить служебные инструкционные файлы
```

## 7. Как это связано с AI-агентом (opencode)

На практике артефакты редко пишут руками — их пишет агент, а CLI даёт ему шаблоны и проверяет результат. В этом репозитории есть готовые обёртки под opencode (`.opencode/commands/` и `.opencode/skills/`), которые ложатся ровно на флоу выше:

| Шаг флоу | Команда в opencode |
|---|---|
| Подумать над идеей | `/opsx-explore` |
| Создать change + все артефакты | `/opsx-propose` |
| Реализовать задачи | `/opsx-apply` |
| Поправить план и сохранить когерентность | `/opsx-update` |
| Влить дельты в основные спеки | `/opsx-sync` |
| Заархивировать | `/opsx-archive` |

CLI — это «двигатель» (данные, инструкции, проверки), а команды/скиллы — «руль» для агента.

## 8. TL;DR — быстрый старт

```bash
# 1. один раз на проект
openspec init

# 2. новое изменение
openspec new change add-two-factor
openspec status --change add-two-factor          # что делать дальше

# 3. по каждому артефакту: взять инструкции и написать файл
openspec instructions proposal --change add-two-factor
# ...пишем proposal.md, затем specs(дельта), design, tasks...

# 4. проверить
openspec validate add-two-factor --strict

# 5. реализовать задачи из tasks.md (отмечая [x])

# 6. закрыть цикл: влить дельты в specs/ и заархивировать
openspec archive add-two-factor
```

**Главная мысль:** сначала договариваемся *что и как меняем* (change с артефактами), потом пишем код, потом возвращаем изменение в живую спеку. Спека всегда отражает реальность, а агент всегда работает по понятным рельсам.
